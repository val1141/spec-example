Сценарий отображения индикатора непрочитанных сообщений на кнопке раздела "Сервисы" в таббаре
1

1. Запросы к API

(upd. 13.11.2024 запросы делаются только если статус пользователя active)

При каждом открытии дашборда (или PTR на этой странице) выполняем запрос к grpc-api-gateway:

  • GET /api/v1/notification/count
    ?filter.orpons=V3UserDto.V3UserBuildingDto[0].orpon_id
    &filter.roomNumbers=V3UserDto.V3UserBuildingDto[0].V3UserEntranceDto[0].V3UserFloorDto[0].V3UserFlatDto[0].name
    (upd. 13.11.2024 исправлена ошибка, было указано V3UserFloorDto[0].name)
    &filter.statusTypes=NEW
    &filter.statusTypes=DELIVERED

ОРПОН и номер квартиры пользователя подставляем из ответа метода GET /api/v3/app/users/current сервиса Household

2. Отображение индикатора

Если метод GET /api/v1/notification/count вернул больше 0 на кнопке раздела "Сервисы" в таббаре отображаем красную точку - индикатор. 

Индикатор должен сохраняться при дальнейшей навигации пользователя по приложению.



Дополнение к сценарию просмотра сервисов
2

1. UI

2. Запросы к API

При каждом открытии раздела "Сервисы" (или PTR на этой странице) повторно выполняем запрос к grpc-api-gateway:

  • GET /api/v1/notification/count
    ?filter.orpons=V3UserDto.V3UserBuildingDto[0].orpon_id
    &filter.roomNumbers=V3UserDto.V3UserBuildingDto[0].V3UserEntranceDto[0].V3UserFloorDto[0].V3UserFlatDto[0].name
    (upd. 13.11.2024 исправлена ошибка, было указано V3UserFloorDto[0].name)
    &filter.statusTypes=NEW
    &filter.statusTypes=DELIVERED

ОРПОН и номер квартиры пользователя подставляем из ответа метода GET /api/v3/app/users/current сервиса Household

3. Отображение индикатора

Если метод GET /api/v1/notification/count вернул больше 0 на кнопке раздела "Уведомления" в топбаре отображаем красную точку - индикатор.

Сценарий отображения уведомлений


1. Запросы к API

Для отображения страницы выполняем запросы:

  • (при необходимости) GET /api/v1/notification/count
    ?filter.orpons=V3UserDto.V3UserBuildingDto[0].orpon_id
    &filter.roomNumbers=V3UserDto.V3UserBuildingDto[0].V3UserEntranceDto[0].V3UserFloorDto[0].V3UserFlatDto[0].name (upd. 13.11.2024 исправлена ошибка, было указаноV3UserFloorDto[0].name)
  • GET /api/v1/notification/list
    ?filter.orpons=V3UserDto.V3UserBuildingDto[0].orpon_id
    &filter.roomNumbers=V3UserDto.V3UserBuildingDto[0].V3UserEntranceDto[0].V3UserFloorDto[0].V3UserFlatDto[0].name

ОРПОН и номер квартиры пользователя подставляем из ответа метода GET /api/v3/app/users/current сервиса Household.

API по умолчанию возвращает уведомления отсортированные по дате создания, от большего к меньшему.

2. Карточка увдомления

см. фигму

Из каждого объекта Notification вернувшегося в ответ на вызов метода GET /api/v1/notification/list отрисовываем карточку уведомления:

Название сервиса к которому относится уведомление - Notification.appTitle (appTitle не является обязательным параметром, если он не пришёл отображаем текст "Уведомление").

Заголовок уведомления - Notification.header

Тест уведомления - Notification.text

Дата создания уведомления - Notification.createdAt

Индикатор что уведомление не прочитано отображаем на объектах, где Notification.statusType != VIEWED

3. Группировка уведомлений

Уведомления на странице отображаются в порядке от более нового к более старому, отсортированными по дате их создания (Notification.createdAt).

Уведомления объединяются в группы по дню и году создания, согласно следующим правилам:

  1. Группа Сегодня: все уведомления, созданные в текущий день (дата создания уведомления совпадает с сегодняшней датой).

  2. Группа Вчера: уведомления, созданные вчера (дата создания уведомления соответствует предыдущему дню).

  3. Группа по дате в текущем году: если уведомление было создано раньше, чем вчера, но в пределах текущего года: название группы отображается в формате: «число месяц» (например, "10 октября").

  4. Группа по дате в прошлом году и ранее: если уведомление было создано в предыдущем году или ранее название группы отображается в формате: «число месяц год» (например, "10 октября 2023").

4. Кнопка "Прочитать все"

Кнопка предназначена для перевода всех уведомлений пользователя в статус VIEWED.

При нажатии вызывает метод POST /bell/api/v1/notification/view/orpon/{V3UserDto.V3UserBuildingDto[0].orpon_id}/room_number/{V3UserDto...V3UserFlatDto[0].name}.

После нажатия на данную кнопку должен быть обновлён статус индикации.

Если метод "Прочитать всё" вернул ошибку никаких действий не производим.

Троттлинг нажатия на кнопку составляет 3 секунды или кнопка дизейблится до получения ответа на связанный с ней запрос.

Делаем по отдельной задаче, для работы кнопки сделаем отдельный метод на бэке, в данный момент кнопка не активна.

5. Пагинация

На странице делаем инфинит скролл. Загружаем по 20 элементов.

6. Фичетоггл

Фича привязана к фичетоглу страницы "Сервисы".

7. Зероскрин

Если уведомлений нет используем стандартный зероскрин с текстом "Пока уведомлений нет".

Для остальных случаев используем соответствующие стандартные зеро скрины. 

8. PTR

Нужен.



Сценарий перехода по уведомлению



Переход по уведомлению

При нажатии пользователем на уведомление:

  • на уведомлении отображаем лоадер (https://pixso.net/app/editor/20EKX827os_LNGF4XgFeHw?icon_type=1&page-id=11294%3A12936);
  • инициируем сценарий перехода как на странице сервисов полностью аналогично сценарию на странице "Сервисы", дополнительно добавляя к ссылке query параметры из Notification.payload.

Лоадер отображается пока не пришли ответы от запросов:

  • GET /multiapp/api/v1/app_lite/{id}
  • GET /multiapp/api/v1/app/{AppLite.id}/code

При отображении лоадера уведомления не кликабельны.

Приложение для перехода определяем по ID указанному в параметре Notification.payload.app_id, вызвав метод GET /multiapp/api/v1/app_lite/{Notification.payload.app_id}.

Если параметр уведомления не равен VIEWED, вне зависимости от результата выполнения перехода, по отношению к такому уведомлению выполняем метод: POST /bell/api/v1/notification/{id}/view.

Ошибки изменения статуса уведомления пользователю не отображаем.

Процесс перехода запускаем не дожидаясь ответа метода POST /bell/api/v1/notification/{id}/view и вне зависимости от результата выполнения этого запроса.

Отображаем ошибку "Сервис партнёра недоступен" стандартными средствами (алерт вида появляющийся сверху) в случае, если:

  •  приложение с Notification.payload.app_id не найдено или метод GET /multiapp/api/v1/app_lite/{id} вернул ошибку;
  •  параметр Notification.payload.app_id отсутствует;
  •  параметр App.statusType полученного приложения не равен ACTIVE;
  •  метод GET /bell/api/v1/notification/{id} вернул ошибку или пустой ответ.

2. При нажатии на уведомление:

  • открывается страница "Уведомления" (для PWA запущенного на IOS дальнейшие не выполняются);
  • пользователю отображается стандартный лоадер;
  • уведомлению, вне зависимости от результата выполнения перехода, устанавливается статус "VIEWED" методом POST /bell/api/v1/notification/{id}/view;
  • вызывается метод получения уведомления по id GET /bell/api/v1/notification/{id} уведомления из push сообщения (notification_id)}
  • выполняется сценарий перехода страницы "Сервисы";
  • к ссылке добавляем в качестве query параметров все поля из поля Notification.payload;
  • в запросах связанных с обработкой пуша, в заголовок X-REQUEST-ID передаётся data.request_id
